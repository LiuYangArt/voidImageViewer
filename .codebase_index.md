# voidImageViewer 代码库索引

本索引面向需要快速进入仓库的开发者或 AI 协作代理。目标不是替代源码，而是减少无效扫描，帮助你先找到正确入口，再读对应代码。

## 1. 项目摘要

- 项目：`voidImageViewer`
- 类型：原生 Windows 图片查看器
- 主要语言：`C`
- UI 技术：`Win32 API` + `.rc` 资源脚本
- 图片能力：BMP / GIF / ICO / PNG / JPG / TIF / WEBP，含 GIF / WEBP 动画
- 当前推荐工程：`vs2026/voidImageViewer.sln`

## 2. 顶层目录

| 路径 | 作用 | 备注 |
| --- | --- | --- |
| `src/` | 主程序源码 | 绝大多数行为都在这里 |
| `res/` | 资源脚本、图标、版本资源 | 选项页、菜单、对话框在此定义 |
| `vs2026/` | 首选 Visual Studio 工程 | 当前优先修改 |
| `vs2019/` | 兼容旧版 VS 工程 | 新增文件时通常也要同步 |
| `vs2005/` | 历史工程 | 非必要不优先动 |
| `libwebp/` | 第三方 WebP 源码 | 工程直接编译其源码 |
| `nsis/` | 安装器相关 | 发布或安装器任务再看 |
| `docs/plans/` | 计划文档 | 现有 PSD 支持计划在这里 |

## 7. 热点与风险点

### `src/viv.c` 是绝对热点

- 文件极大，包含多种职责
- 不能先入为主地做全文件重构
- 处理方式应该是“关键词搜索 -> 只读相关函数 -> 小范围改动”

### 资源改动通常是联动改动

如果用户可见界面有变化，往往需要联动检查：

- `res/resource.h`
- `res/voidImageViewer.rc`
- `src/localization.h`
- `src/localization_en_us.h`
- `src/localization_zh_cn.h`
- `src/viv.c`

### 新文件通常要同步两个工程

如果新增 `.c` / `.h` / 资源文件，通常至少同步：

- `vs2026/voidImageViewer.vcxproj`
- `vs2019/voidImageViewer.vcxproj`

## 8. 已知仓库事实

- `vs2026/voidImageViewer.vcxproj` 中列出了 `render.h`、`render_d3d.h`、`render_gdi.h`、`render_opengl.h`、`render_software.h`，但这些头文件当前不在 `src/` 中，源码中也没有实际 `#include`，可视为陈旧工程引用。
- `libwebp/` 下存在 `PaxHeaders.X/` 目录，属于第三方归档残留；除非任务明确要求，不要顺手清理。
- `docs/plans/2026-04-01-psd-support-plan.md` 已记录一份 PSD 支持方案，可作为后续相关任务的设计上下文。

## 9. 推荐阅读粒度

为了减少无效上下文，建议按下面粒度读取文件：

- `src/viv.c`: 只读目标函数、调用点和相邻辅助逻辑
- `res/voidImageViewer.rc`: 只读相关对话框、菜单或版本块
- `src/localization*.h`: 只读相关 ID 附近条目
- `vs2026/voidImageViewer.vcxproj`: 只读相关配置段或文件清单段

## 10. 一句话总结

这个仓库的真实工作方式是：

- 大部分产品行为在 `src/viv.c`
- 特殊格式解码走独立模块范式，`src/webp.c` 是样板
- 用户可见功能改动通常联动资源、本地化、配置和工程文件
- 修改应以小步、定点、可验证为原则
